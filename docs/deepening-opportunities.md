## 1. Tag Serialization Spread Across DB Modules

**Files:** `src/lib/musicas-db.ts`, `src/lib/eventos-db.ts`, `src/lib/setlists-db.ts`, `src/lib/practice-sessions-db.ts`

**Problem:** Every DB module independently handles tag serialization. `JSON.parse(row.tags || '[]')` appears at least 12 times across the codebase. If the schema changes (e.g., `null` vs `'[]'` handling, or a different serialization format), you'd need to update all of them. This is **low locality** — the knowledge of "how to serialize tags" is spread everywhere instead of living in one place.

**Solution:** Extract tag serialization into a dedicated module (e.g., `tag-serializer.ts`) with `serialize()` and `deserialize()` functions. The DB modules become clients of this module.

**Benefits:**
- **Locality**: One place to change if tag format evolves
- **Leverage**: Any future tag-related logic (validation, merging, filtering) lives here
- **Tests**: Easy to test serialization edge cases in isolation

## 2. Type Drift Between `types/database.ts` and DB Modules

**Files:** `src/types/database.ts` vs `src/lib/musicas-db.ts`, `src/lib/eventos-db.ts`, `src/lib/setlists-db.ts`

**Problem:** `Musica` has **two different shapes** depending on which file you look at:
- `types/database.ts`: `tom_original`, `cifra`, `tags`, no `observacao`, no `drum_pattern_id`
- `musicas-db.ts`: Has `observacao`, `audio_url`, `groove`, `drum_pattern_id`, `bpm`, `volume`

The `types/database.ts` version is incomplete — it's not the source of truth. Every API route and component using `types/database.ts` is working with a stale view of the actual domain.

**Solution:** Make `musicas-db.ts` (and similar) **the** source of truth for domain types, and have `types/database.ts` import from them. Or consolidate into a single `domain/` module.

**Benefits:**
- **No more drift**: One definition, not three
- **Locality**: Bugs from type mismatches concentrate in one place
- **Leverage**: Refactoring the domain type is one edit, not four

## 3. ChordViewer Owns Duplicate Transposition Logic

**Files:** `src/utils/chord-transposer.ts` vs `src/components/chords/ChordViewer.tsx`

**Problem:** `ChordViewer.tsx` contains its own implementations of:
- `isChordProFormat()`
- `isTabLine()`
- `isStrummingLine()`

These are duplicated from `chord-transposer.ts`. The transposition logic itself lives in `transposeLine()` / `transposeCifra()`, but the **parsing** logic (detecting what kind of line this is) is duplicated. This means bugs in tablature detection could be fixed in one place but remain in another.

**Solution:** Move all chord/line detection and transposition logic into a single **Chordpro parser module** behind an interface. Both `ChordViewer` and `CifraViewer` become thin clients of this module.

**Benefits:**
- **Locality**: Bug fixes and enhancements to chord/tab detection happen once
- **Leverage**: A test of the parser module exercises the full logic surface
- **Deepening**: The module earns its keep — it contains real knowledge about how to parse cifra formats

## 4. `musicasDb` is Absorbing Wider Concerns

**Files:** `src/lib/musicas-db.ts`

**Problem:** The `Musica` data access module has organically grown to include `drum_pattern_id`, `bpm`, `volume` — which are really **performance settings**, not core song metadata. These concerns are orthogonal to "a song exists in the catalog."

Deletion test: if you deleted `musicas-db.ts` but needed to change how `bpm` is stored or validated, would the complexity disappear or would it reappear across call sites? It would **reappear** — meaning the module isn't actually hiding complexity, it's just moving it.

**Solution:** Consider whether `bpm`, `volume`, and `drum_pattern_id` belong in a separate **PerformanceSettings** module (or attached to the EventoMusica join, since those settings might be per-event, not per-song).

**Benefits:**
- **Locality**: Performance settings bugs fix in one place
- **Leverage**: If settings vary per-event (transposition does), this model is more accurate
- **Tests**: Smaller surface to test per-song vs per-event behavior

## 5. API Routes Are Thin Pass-Throughs

**Files:** `src/app/api/musicas/[id]/route.ts` and 30+ others

**Problem:** API routes do request parsing → call DB module → return JSON. There's no **service layer** to hold business logic. This means:
- Validation is ad-hoc and duplicated
- Composite operations (e.g., "create evento from template + copy musicas + set initial status") are either duplicated or handled by one route that shouldn't own that knowledge

**Solution:** Introduce a service layer module between API routes and DB modules. API routes become thin adapters.

**Benefits:**
- **Locality**: Business rules have a home
- **Leverage**: Common validation (e.g., "can't add a music to a canceled event") lives once
- **Tests**: Service layer is easy to unit test without HTTP

## Glossary

- **Module** — anything with an interface and an implementation (function, class, package, slice)
- **Interface** — everything a caller must know to use the module: types, invariants, error modes, ordering, config
- **Depth** — leverage at the interface: a lot of behaviour behind a small interface
- **Seam** — where an interface lives; a place behaviour can be altered without editing in place
- **Leverage** — what callers get from depth
- **Locality** — what maintainers get from depth: change, bugs, knowledge concentrated in one place
